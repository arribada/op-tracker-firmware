9 /**
10  * @page kns_q_page OS queue utilities
11  *
12  * This page is presenting the utilities to manipulate queues used in kineis_sw.
13  *
14  * @section kns_q_intro Introduction
15  *
16  * As on lots of OS, queues are used by kineis SW environement (kineis stack + application) to
17  * share events and data between different tasks/ISR contexts. Source SW entity (task) pushes
18  * event+data into the queue from one context (e.g. ISR) and the destination layer (task) will pop
19  * it from its own context.
20  *
21  * @section kns_q_list List of queues
22  *
23  * Kineis OS does not assume any priorities between tasks directly. But it is done through the side
24  * effect of priorities declared on queues. An event delivered in a high-priority queues shall be
25  * consumed as soon as possible by the dedicated task.
26  *
27  * From global kineis SW architecture, below is some example of queues used between ISR/TASK
28  * contexts. It is sorted from lowest to highest priority order:
29  * * 3: KNS_Q_DL_APP2MAC used from application to Kinéis stack (MAC internal component)
30  * * 2: KNS_Q_UL_MAC2APP used from Kinéis stack to application (MAC internal component)
31  * * 1: KNS_Q_UL_SRVC2MAC internally used internally in Kineis stack, to send end-of-processing
32  * events from RF DRIVER ISRs to task context layer (MAC internal component)
33  *
34  * @section kns_q_design Design
35  *
36  * As kineis_sw is fully known, queues above are hardcoded and statically defined as arrays.
37  *
38  * The size and content of each queue shall be defined through kns_q.h and kns_q.c. Refer to
39  * @ref kns_q_conf_page for some extra details
40  *
41  * There is write and read pointer to the cells of the array. Roll-over is handled. Check on FIFO
42  * full and FIFO empty is done.
43  *
44  * @attention due to simple FIFO-full mechanism implemented so far, queues needs to be one element
45  * longer than expected by the system, in case of baremetaol Kineis OS.
46  *
47  * A priority is assigned to those queues, meaning some event in higher-priority queue should be
48  * executed before lower-priority queue's events (refer to kineis_sw architecture for details).
49  *
50  */

74 /**
75  * @brief This function is used to create queue
76  *
77  * This function is aimed at creating the queue used in KNS libraries.
78  * Size and length of the queues are statically defined KNS_Q_conf.h.
79  *
80  * This function also handles the correspondence between kineis queue handlers (KNS_Q_handle_t) and
81  * RTOS handlers, so that KNS_Q_push/pop can still be used with kineis queue handler references.
82  *
83  * @note This routine is only useful in case of RTOS. In internal baremetal kineis OS, those queues
84  * are directly defined as uint8t arrays during compilation. In case of external RTOS, we need to
85  * invoke queue creation routine instead.
86  *
87  * @param[in] qHandle: kineis queue handle
88  * @param[in] qLength: length of the queue in number of element
89  * @param[in] qEltByteSize: length of each element of the queue (in bytes).
90  * @retval KNS_STATUS_OK if creation was ok, KNS_STATUS_ERROR otherwise.
91  */
92 enum KNS_status_t KNS_Q_create(enum KNS_Q_handle_t qHandle, uint8_t qLength, uint16_t qEltByteSize);
